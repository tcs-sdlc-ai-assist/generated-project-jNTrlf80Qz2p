## 1. System Overview

### 1.1 System Identity & Purpose

The ChatApp Demo is a fully client-side, single-page application (SPA) that simulates a real-time chat experience entirely within the browser. It is designed as a static-frontend demonstration ŌĆö not a production chat system ŌĆö with the explicit goal of showcasing core chat UX patterns: presence awareness, typing indicators, delivery and read receipts, emoji integration, and multi-participant conversations. The system has no backend server, no real database, and no external service dependencies. Every operation ŌĆö authentication, message persistence, real-time event propagation ŌĆö is executed within the browser runtime using only the Web Platform APIs available to any modern browser: `localStorage` for durable storage and the `BroadcastChannel` API for cross-tab communication.

The application serves two primary purposes. First, it is an evaluative tool: stakeholders can open the application in one or more browser tabs, log in as different mock users, and immediately experience a realistic chat interaction without any infrastructure setup. Second, it is an architectural reference: the codebase demonstrates how a disciplined in-memory repository pattern, a well-structured event bus over BroadcastChannel, and a layered React component tree can collectively deliver a responsive, real-time-feeling application with zero backend cost.

The system's identity is defined by its constraints as much as its capabilities. It is intentionally limited to the browser origin's security sandbox. It stores all data ŌĆö user credentials, conversation metadata, message bodies, delivery status records ŌĆö in `localStorage`, a synchronous key-value store with a typical 5ŌĆō10 MB quota per origin. It simulates real-time messaging not through WebSockets or Server-Sent Events, but through the BroadcastChannel API, which allows same-origin browsing contexts (tabs, iframes) to exchange messages on a named channel. This design choice means the "real-time" experience is confined to a single browser instance; two users on separate machines cannot chat with each other. This limitation is by design and is clearly communicated to evaluators through strategic UI hints embedded in the conversation header and settings screen.

### 1.2 Execution Model & Runtime Characteristics

The ChatApp Demo executes as a pure client-side SPA. When a user navigates to the deployed URL, Vercel's static hosting serves a pre-built bundle of HTML, CSS, and JavaScript. From that point forward, all computation happens in the browser's main thread. There is no server-side rendering (SSR), no API gateway, no middleware layer, and no database server. The application's "backend" is the set of in-memory data structures and the `localStorage` persistence layer that backs them.

The execution model follows a strict initialization sequence. On page load, the application performs a synchronous feature-detection check for `localStorage` and `BroadcastChannel` availability. If `localStorage` is unavailable or disabled, the application renders a full-screen message instructing the user to enable it ŌĆö the system cannot function without durable storage. If `BroadcastChannel` is unavailable (rare in modern browsers but possible in older versions or restricted contexts), the application degrades gracefully to single-tab mode, disabling cross-tab real-time features while preserving all local functionality: message sending, conversation browsing, and emoji insertion all continue to work.

Once the environment check passes, the application reads the `currentUser` session flag from `localStorage`. If a valid session exists, the user is routed directly to the main chat interface. If not, the authentication screen is rendered. On first-ever load ŌĆö when `localStorage` contains no `users` key ŌĆö the auto-seeding logic executes synchronously, populating 3ŌĆō5 demo user accounts, 2ŌĆō3 pre-built conversations, and 5ŌĆō15 sample messages per conversation. This seeding ensures the application is immediately demonstrable with zero manual setup, directly supporting the activation-rate success metric (Ōēź60% of demo users sending at least one message).

The runtime is single-threaded by virtue of JavaScript's event loop, but the application employs asynchronous patterns for non-blocking operations where possible. `localStorage` reads and writes are synchronous by API design, which means the repository layer must be disciplined about batching writes and avoiding large synchronous reads on the critical rendering path ŌĆö particularly during conversation-list loading, which must complete in under one second. The BroadcastChannel API is asynchronous: messages posted to the channel are delivered to other browsing contexts via the event loop, introducing a small but predictable latency (typically under 200ms within the same browser instance). This latency is well within the real-time responsiveness budget and is imperceptible to users in the multi-tab demo scenario.

Memory management is a deliberate concern. The in-memory repository holds all conversation and message data for the current session in JavaScript Map structures. For a demo application with pre-seeded data and moderate usage, this is well within the memory capacity of any modern browser. However, the architecture anticipates the `localStorage` quota limit (~5ŌĆō10 MB) and includes a quota monitoring mechanism that warns users when usage exceeds approximately 80% of the estimated quota, with a fallback strategy of trimming oldest messages if writes begin to fail.

## 2. Architecture Style & Pattern Selection

### Pattern Selection: Client-Side-Only SPA

The ChatApp Demo adopts a **Client-Side-Only Single Page Application (SPA)** architectural pattern. This is not merely a preference ŌĆö it is the defining constraint of the system, driven by the explicit requirement that the application must operate with no backend server, no real database, and no external service dependencies. Every operation ŌĆö authentication, message persistence, real-time event propagation, search ŌĆö executes entirely within the browser runtime. The architecture embraces the browser as the sole compute, storage, and networking substrate.

The SPA pattern is characterized by a single HTML document shell loaded from static hosting (Vercel), with all subsequent navigation and rendering handled by JavaScript executing in the browser. Vite serves as the build toolchain, producing a static bundle of HTML, JavaScript, and CSS assets with content-hashed filenames for cache-busting. React JS provides the component model and declarative UI rendering, managing the DOM through its virtual DOM reconciliation algorithm. Tailwind CSS delivers utility-first styling compiled to a minimal production stylesheet during the Vite build. Client-side routing manages navigation between the login screen (`/login`), main chat interface (`/chat`), and settings screen (`/settings`) without page reloads, using the History API under the hood.

Why is the SPA pattern the correct choice here? First, it aligns perfectly with the deployment constraint: Vercel static hosting serves pre-built assets with no server-side execution. There is no Node.js process, no API server, no middleware layer ŌĆö just static files delivered over a global CDN. The entire application is a collection of immutable assets that can be deployed with a single command and rolled back instantly. Second, the SPA model concentrates all application state in the browser, which is exactly where this demo needs it. User sessions, conversation data, message histories, typing indicator state, presence information, and read receipt tracking all live in the JavaScript heap and localStorage. There is no server-side state to synchronize with, no network latency to account for, and no distributed consistency challenges to solve. The browser is both the application server and the database. Third, the SPA pattern enables the multi-tab demo scenario naturally: each tab is an independent SPA instance communicating with peer tabs through the BroadcastChannel API, with localStorage serving as the shared persistence layer. This architecture turns the browser into a miniature distributed system ŌĆö multiple isolated JavaScript runtimes (tabs) sharing state through a message bus (BroadcastChannel) and a common data store (localStorage).

The trade-off is clear and accepted: this architecture sacrifices multi-device persistence, real backend security, production-grade scalability, and cross-browser synchronization in exchange for zero-infrastructure simplicity, instant deployability, and a self-contained demo experience. These sacrifices are explicitly documented as out-of-scope. The architecture does not attempt to simulate or approximate a backend ŌĆö it embraces the browser as the sole runtime and designs all patterns around that reality. This honesty is architecturally important: by not pretending to have a backend, the system avoids leaky abstractions, fake network latency simulations, and the complexity of maintaining the illusion of a client-server architecture.

A critical implication of the client-side-only SPA pattern is that **all data is ephemeral from the system's perspective**. localStorage is tied to the browser and origin; clearing browser data destroys all application state. This is not a bug ŌĆö it is a feature of the demo architecture, clearly communicated to users through UI hints ("Data stored in your browser's localStorage only. Clearing browser data will reset the app."). The architecture accepts this limitation and designs around it by making the demo immediately re-seedable: on first load with empty localStorage, the app auto-populates demo users, conversations, and messages, ensuring the experience is always recoverable. The auto-seeding logic is idempotent ŌĆö it runs once, gates on an empty `users` key, and never overwrites existing data.

The SPA's initialization sequence establishes the architectural rhythm for the entire application. On page load: (1) the static HTML shell loads from Vercel's CDN; (2) the React application bootstraps and performs a localStorage feature-detection check ŌĆö if localStorage is unavailable, a full-screen message is displayed; (3) the auto-seeding logic checks for an empty `users` key and populates demo data if needed; (4) the auth service reads the `currentUser` session flag from localStorage; (5) if absent, the router directs to `/login`; if present and valid (references an existing user), the router directs to `/chat`; (6) the repository layer hydrates its in-memory stores from localStorage, deserializing JSON into entity objects; (7) the BroadcastChannel service opens the `chat-app-demo` channel and begins listening for events from peer tabs; (8) the UI renders the conversation list and awaits user interaction. This sequence is entirely synchronous from the user's perspective ŌĆö there are no network round-trips, no loading spinners for data fetches, no API authentication handshakes. The architecture delivers a fast, frictionless startup that serves the demo's goal of maximizing time spent interacting with chat features rather than waiting for infrastructure.

### Internal Layering Strategy (Repository ŌåÆ Service ŌåÆ UI)

Although the ChatApp Demo runs entirely within a single browser runtime, the architecture enforces a strict three-layer decomposition to maintain separation of concerns, testability, and maintainability. The layers are: **Repository Layer** (data access and integrity), **Service Layer** (business logic and orchestration), and **UI Layer** (presentation and user interaction). Data flows downward through these layers; events flow upward. This is not an arbitrary decomposition ŌĆö it is a deliberate application of the Dependency Inversion principle adapted for a client-side-only runtime: higher layers depend on lower layers through well-defined interfaces, and lower layers never depend on higher layers.

**Repository Layer.** This is the foundation of the architecture. The repository layer owns all data access logic and serves as the single source of truth for the application's domain entities: User, Conversation, ConversationMember, Message, and MessageDelivery. Each entity has a corresponding repository module that maintains an in-memory data structure (a `Map` keyed by entity ID for O(1) lookup) and synchronizes every mutation to localStorage. The pattern is **write-through**: every create, update, or delete operation first modifies the in-memory store, then immediately serializes the entire collection to localStorage under the appropriate key (`users`, `conversations`, `messages`, `messageDeliveries`). Reads are served exclusively from the in-memory store, ensuring zero-latency access after initial hydration. The repository layer exposes a synchronous API ŌĆö there are no Promises, no async/await, no callbacks ŌĆö because localStorage access is synchronous by nature and the data volumes are small enough that `JSON.stringify` overhead is negligible (sub-millisecond for typical payloads of a few hundred messages). This synchronous API simplifies every layer above it: services and UI components can call repository methods without managing async state, loading indicators, or error boundaries for failed fetches.

The repository layer also enforces data integrity constraints at the lowest possible level. For example, the ConversationRepository's `createDirect(userA, userB)` method checks for an existing DM between the two users before creating a new conversation, preventing duplicates at the data layer rather than relying on UI-level checks that could be bypassed. The MessageRepository enforces the 4,000-character limit and rejects blank or whitespace-only messages with structured error objects. The MessageDeliveryRepository enforces monotonic progression of `last_read_message_id` ŌĆö the ID can only move forward, preventing race conditions where an older read receipt overwrites a newer one. These constraints are implemented once, in the repository, and are guaranteed to be enforced regardless of which code path triggers the operation ŌĆö whether from the UI, from a BroadcastChannel event handler, or from the auto-seeding logic.

**Service Layer.** The service layer orchestrates business logic that spans multiple repositories or involves cross-cutting concerns like BroadcastChannel event publication. The key services map directly to the epic boundaries: `mockAuth` (addresses SCRUM-20951 ŌĆö registration, login, logout, session management, demo user seeding), `mockConversations` (conversation CRUD, member management, admin role enforcement), `mockMessages` (message sending pipeline, read receipt progression, typing indicator lifecycle), and `BroadcastChannelService` (channel lifecycle, event serialization/deserialization, heartbeat management, graceful degradation). Each service depends on one or more repositories and may depend on other services (e.g., `mockMessages` depends on `BroadcastChannelService` to publish `message.created` events, and `mockConversations` depends on `mockMessages` to retrieve last message previews).

The service layer is where the architectural value of layering becomes most apparent. Consider the message sending pipeline ŌĆö the most critical data flow in the system. The UI calls `mockMessages.send(convId, senderId, content)`. The service executes a multi-step orchestration: (1) validate the content (non-blank, within 4,000 characters) ŌĆö if invalid, return a structured error immediately; (2) call `MessageRepository.create()` to persist the message entity with a generated ID, timestamp, and initial status of `sent`; (3) call `ConversationRepository.updateLastMessageAt()` to bump the conversation's sort timestamp, ensuring it rises to the top of the conversation list; (4) for each recipient in the conversation, call `MessageDeliveryRepository.create()` to initialize a delivery record with status `sent`; (5) call `BroadcastChannelService.publish('message.created', serializedPayload)` to notify peer tabs; (6) return the created message entity to the UI for immediate optimistic display. This orchestration involves three repositories and one cross-cutting service. If this logic were embedded in a React component's event handler, it would be untestable in isolation, unreusable across different UI contexts (e.g., sending from a toast reply vs. the main composer), and tightly coupled to the component lifecycle. By extracting it into the service layer, the orchestration becomes a pure function of its dependencies, testable with mock repositories, and reusable across any UI component that needs to send a message.

The service layer also owns the BroadcastChannel event consumption pipeline. When a `message.created` event arrives from a peer tab, the `BroadcastChannelService` deserializes the payload and routes it to `mockMessages.handleIncomingMessage(payload)`. This method calls the same `MessageRepository.create()` and `ConversationRepository.updateLastMessageAt()` methods used for locally-created messages, ensuring that all messages ŌĆö regardless of origin ŌĆö flow through the identical persistence pipeline. The UI does not distinguish between "my message" and "their message" at the data layer; the only difference is the `sender_id` field. This design eliminates an entire class of bugs where locally-created and remotely-received messages would follow different code paths with subtly different behavior.

**UI Layer.** The UI layer is composed of React components organized by feature domain: Auth (LoginForm, RegistrationForm, DemoUserButtons), Conversation (ConversationList, ConversationHeader, NewConversationModal, GroupSettingsPanel), Message (MessageThread, MessageBubble, DateSeparator, MessageComposer, EmojiPicker, TypingIndicator), Presence (PresenceDot, LastSeenLabel), and Settings (SettingsScreen, ReadReceiptsToggle). Components consume services through React hooks and context providers ŌĆö never through direct import of service modules. The `useAuth` hook exposes `user`, `login`, `register`, `logout`, and `isAuthenticated` by wrapping `mockAuth`. The `useConversation` hook exposes conversation list state and operations (`createDirect`, `createGroup`, `addMember`, `removeMember`) by wrapping `mockConversations`. The `useMessages` hook exposes message thread state, the `send` function, and typing indicator state by wrapping `mockMessages`. The `usePresence` hook exposes online/offline status for all users by subscribing to the PresenceRepository.

Critically, UI components never access localStorage directly and never interact with the BroadcastChannel API directly. All data access flows through the repository layer via the service layer. All real-time events are consumed by the service layer, which updates the in-memory repositories, which triggers React re-renders through the hook/context mechanism. This ensures that the UI remains a pure reflection of application state, with no hidden side effects or direct browser API dependencies. A React component rendering a message thread does not know or care whether the messages came from localStorage hydration, a local send operation, or a BroadcastChannel event ŌĆö it simply renders whatever the repository contains.

The layering strategy also defines the failure boundary: if localStorage is full or unavailable, the repository layer detects the `QuotaExceededError` and throws a typed error. The service layer catches this error and translates it into a structured error state (e.g., `{ type: 'STORAGE_FULL', message: '...' }`). The UI layer receives this error state through the hook and renders the quota warning banner. At no point does a React component catch a `QuotaExceededError` directly or inspect the `localStorage.remainingSpace` property. This separation keeps error handling logic centralized, consistent, and testable independently of the UI.

### BroadcastChannel as Real-Time Event Bus ŌĆö Pattern Justification

The BroadcastChannel API is the architectural centerpiece of the real-time simulation, directly addressing the core messaging epic (SCRUM-20950). It enables multiple browser tabs sharing the same origin to exchange messages without a server, forming a peer-to-peer event bus within the browser. The ChatApp Demo opens a single BroadcastChannel named `chat-app-demo` on application initialization and uses it to propagate six event types: `message.created`, `message.status` (delivered/read transitions), `typing.start`, `typing.stop`, `presence.update`, and `conversation.updated` (rename, membership changes).

The architectural pattern is **publish-subscribe (pub/sub) over a shared named channel**. Each browser tab is simultaneously a publisher and a subscriber. When a user sends a message, the sending tab publishes a `message.created` event containing the serialized message payload ŌĆö including the message entity, conversation ID, and sender metadata. All other tabs subscribed to the `chat-app-demo` channel receive the event through their `onmessage` handler, deserialize the payload via the structured clone algorithm, and route it to the appropriate service method (`mockMessages.handleIncomingMessage`). The service updates the in-memory repositories, which triggers React re-renders, causing the message to appear in the recipient's UI. The sending tab does not need to know how many peer tabs exist, which users are logged into them, or what their tab IDs are ŌĆö it simply publishes to the named channel, and the browser runtime handles delivery to all subscribers within the same origin. This is the same architectural pattern used by Redis Pub/Sub, Apache Kafka, and other message brokers, but implemented entirely within the browser's built-in APIs.

Why BroadcastChannel over alternatives? The primary driver is the architectural constraint: no server. WebSockets require a WebSocket server for the initial HTTP upgrade handshake and ongoing connection management. Server-Sent Events require an HTTP server to maintain long-lived connections. WebRTC requires a signaling server for peer discovery and connection establishment (even though the data channel is peer-to-peer). BroadcastChannel requires none of these ŌĆö it is a pure browser-to-browser API scoped to the same origin, requiring zero infrastructure beyond what the browser already provides. This makes it the only viable real-time transport for a fully client-side architecture deployed as static files.

Additionally, BroadcastChannel has several properties that align well with the demo's needs. First, `postMessage()` returns immediately ŌĆö it is non-blocking from the publisher's perspective, so message sends are not delayed by peer tab processing. Second, the structured clone algorithm handles serialization of complex JavaScript objects (including Dates, Maps, Sets, and nested objects) without manual JSON stringification, though the service layer adds an explicit serialization step for consistency and debugging. Third, BroadcastChannel is supported in all modern browsers: Chrome 54+, Firefox 38+, Safari 15.4+, and Edge 79+ ŌĆö covering the target browser matrix specified in NFR-007. Fourth, the API surface is minimal: `new BroadcastChannel(name)`, `channel.postMessage(data)`, `channel.onmessage = handler`, and `channel.close()` ŌĆö making it easy to wrap in a service abstraction and easy to mock in tests.

The `BroadcastChannelService` module encapsulates the raw API and provides several architectural services beyond simple message passing. It manages the channel lifecycle: the channel is opened on application initialization (after successful auth) and closed on tab unload via the `beforeunload` event, ensuring clean teardown. It implements **echo prevention**: each tab generates a random `tabId` on initialization and includes it in every published event; the receiving handler compares the event's `tabId` to its own and discards events originating from the same tab, preventing duplicate processing of locally-initiated actions. It provides **typed event routing**: rather than a single `onmessage` handler with a switch statement, the service maintains a registry of handlers keyed by event type, allowing services to subscribe to specific events (`broadcastChannelService.on('message.created', handler)`) without knowledge of other event types. It implements **graceful degradation**: on initialization, the service performs a feature detection check for `window.BroadcastChannel`; if unavailable, the service becomes a no-op (all publish calls are silently ignored, all subscriptions are never invoked), and the app functions in single-tab mode with a UI hint displayed in the conversation header.

The event payload design follows a consistent envelope pattern across all event types: `{ type: string, payload: object, sender: { userId: string, tabId: string }, timestamp: number }`. The `type` field enables the routing registry. The `payload` field contains the event-specific data (message entity, typing user ID, presence status, etc.). The `sender` object identifies the originating user and tab, enabling echo prevention and UI decisions (e.g., not showing a typing indicator for the current user). The `timestamp` is a `Date.now()` value set by the publisher, enabling receivers to detect and handle out-of-order delivery ŌĆö though in practice, BroadcastChannel delivers messages in FIFO order within a single browser session.

The presence system deserves special architectural attention because it uses BroadcastChannel in a novel heartbeat pattern. Each tab publishes a `presence.update` heartbeat every 5 seconds (via `setInterval` managed by the BroadcastChannelService) containing the logged-in user's ID and a status of `online`. Peer tabs receive these heartbeats and update a `lastSeenTimestamps` map in the PresenceRepository. A user is considered `online` if their last heartbeat was received within 10 seconds (a 2x grace period over the heartbeat interval); otherwise, they transition to `offline` with a "last seen" time derived from the most recent heartbeat timestamp. When a tab is closed gracefully, the `beforeunload` event triggers a final `presence.update` with status `offline`, providing an immediate transition. However, because `beforeunload` is not guaranteed to fire (browser crash, force quit, mobile browser tab suspension), the 10-second heartbeat timeout serves as a fallback liveness detector. This is a pragmatic, eventually-consistent approach to presence ŌĆö it does not guarantee instantaneous offline detection, but it converges to the correct state within 10 seconds of a tab disconnecting, which is acceptable for a demo where all "users" are the same person switching between tabs.

### Rejected Alternatives & Trade-off Analysis

Every architectural decision involves trade-offs. This section documents the alternatives that were seriously considered during the design phase and the rationale for their rejection. Each rejection is grounded in the system's constraints (no backend, static hosting, demo scope) and the quality attributes prioritized for this system (simplicity, deployability, demo-ability over scalability, security, durability).

**Server-Backed SPA (React + Node.js/Express + Database).** The most obvious alternative to a client-side-only SPA is a traditional three-tier architecture: React frontend, Node.js/Express API server, and a PostgreSQL or MongoDB database. This architecture would provide real persistence (data survives browser clears), real authentication (bcrypt-hashed passwords, JWT tokens, session management), real WebSocket-based real-time communication (Socket.io with rooms and namespaces), and multi-device synchronization (log in from any browser and see your conversations). It was rejected for two decisive reasons. First, it violates the core constraint stated in the PRD: "No backend server ŌĆö all operations are mock functions reading/writing localStorage." This is not a negotiable preference ŌĆö it is a hard architectural boundary. Second, even if the constraint were relaxed, a server-backed architecture would dramatically increase the complexity of deployment, requiring a server hosting environment (Vercel serverless functions, Railway, Render, or AWS), database provisioning and connection management, environment variable configuration, CORS setup, and a CI/CD pipeline that handles database migrations. All of this infrastructure overhead would serve a demo whose purpose is to showcase chat UX ŌĆö message composition, typing indicators, read receipts, emoji ŌĆö not infrastructure engineering. The client-side-only SPA achieves the same UX goals with zero infrastructure, which is the correct trade-off for a demo.

**Server-Side Rendering with Next.js.** Next.js offers SSR, static site generation (SSG), incremental static regeneration (ISR), API routes, middleware, and a more opinionated framework structure than Vite + vanilla React. It was rejected because SSR provides no benefit for this application. The app has no SEO requirements ŌĆö it is an authenticated chat interface, not a content site that needs to be indexed by search engines. It has no initial load performance concerns that SSR would solve ŌĆö the JavaScript bundle is small (React + Tailwind + application code, likely under 200KB gzipped), and it loads from Vercel's global CDN with near-instant Time to First Byte. It has no server-side data fetching needs ŌĆö all data is in localStorage, so there is nothing to fetch during SSR; the server would render an empty shell that the client would immediately re-render with data, negating any performance benefit. Next.js API routes could theoretically host the service layer logic, but they would require a Node.js runtime (Vercel serverless functions), violating the "no backend" constraint. Vite + React provides a simpler, more direct path to a static SPA with fewer abstractions and a faster build pipeline.

**Micro-Frontends.** A micro-frontend architecture would decompose the application into independently deployable modules ŌĆö for example, an `auth-mfe` owning the login and registration screens, a `chat-mfe` owning the conversation list and message thread, and a `settings-mfe` owning the settings screen ŌĆö each built and deployed by separate teams using Module Federation. This was rejected as massively over-engineered for a demo application built by a single small team (estimated 2-3 developers). Micro-frontends introduce module federation configuration complexity, shared dependency management, cross-micro-frontend communication challenges (custom events, shared state), CSS isolation requirements, and deployment coordination overhead ŌĆö all for an application whose entire codebase fits comfortably in a single repository and is deployed as a single static bundle. The internal layering strategy (Repository ŌåÆ Service ŌåÆ UI) provides sufficient modularity and separation of concerns without the operational burden of micro-frontends. If the application were to grow into a large-scale production system with multiple independent teams, micro-frontends could be revisited ŌĆö but for a demo, they are architectural overkill.

**IndexedDB Instead of localStorage.** IndexedDB is the browser's more powerful storage API, offering significantly more capacity (typically 50MB to unlimited, vs. localStorage's 5-10MB), better querying capabilities (indexes, cursors, range queries), and an asynchronous access pattern that does not block the main thread. It was rejected for several reasons. First, the demo's data volume is intentionally small ŌĆö pre-seeded data plus moderate usage during a demo session fits comfortably within localStorage's quota, and the quota limitation is actually a feature that triggers the quota warning flow (FR-023). Second, localStorage's synchronous API simplifies the repository layer dramatically ŌĆö no async/await, no transaction management, no version change events, no connection lifecycle. The repository methods are pure synchronous functions that return values directly, which simplifies every consumer up the stack. Third, localStorage is universally supported with an API surface that every web developer knows; IndexedDB has a notoriously complex API that would require a wrapper library (like Dexie.js or idb) to be usable, adding a dependency. If the application were to evolve into a production system with larger data volumes and more complex queries, migrating the repository layer to IndexedDB would be a natural architectural evolution ŌĆö the repository abstraction exists precisely to enable this kind of storage engine swap without affecting the service or UI layers.

**Redux or Zustand for State Management.** External state management libraries were considered to manage the application's client-side state. Redux provides a predictable state container with a unidirectional data flow (actions ŌåÆ reducers ŌåÆ store ŌåÆ selectors ŌåÆ UI). Zustand provides a simpler, hook-based state management API. Both were rejected because the repository layer already serves as the single source of truth for domain data, and React's built-in `useState` and `useContext` are sufficient for UI-specific ephemeral state (which conversation is selected, whether the emoji picker is open, the current composer text, which modal is visible). Adding Redux would introduce boilerplate ŌĆö action types, action creators, reducers, selectors, middleware for side effects ŌĆö without adding architectural value. The repository pattern already provides predictable state updates (every mutation goes through a single code path), and the service layer already provides encapsulated business logic with side effects (BroadcastChannel publication). The architecture achieves unidirectional data flow (UI ŌåÆ Service ŌåÆ Repository ŌåÆ localStorage ŌåÆ React re-render) without a dedicated state management library. The hooks-and-context pattern is sufficient for the application's complexity level.

**Service Workers for Background Sync and Offline Support.** Service workers could provide offline support (caching the application shell and data), background message synchronization, and push notification-like behavior through the Notification API. They were rejected because: (1) the demo has no network requests to intercept or cache ŌĆö the entire application loads from static files and all data is local; (2) BroadcastChannel already provides cross-tab communication, which is the primary real-time use case; (3) service workers add significant complexity ŌĆö registration, lifecycle management (install, activate, update), debugging challenges (requires Chrome DevTools Application panel), and scope restrictions; (4) the demo's scope explicitly excludes push notifications and offline support. If offline support were added in a future phase, service workers would be the appropriate architectural addition, caching the static assets and potentially providing a background sync mechanism for messages composed while offline.

**Polling localStorage `storage` Events for Cross-Tab Communication.** Before BroadcastChannel was standardized (2016-2017), cross-tab communication was typically achieved by writing to localStorage and listening for the `window.addEventListener('storage', ...)` event, which fires in all tabs except the one that made the change. This approach was rejected because: (1) the `storage` event only fires in other tabs ŌĆö the writing tab must handle its own "broadcast" separately, leading to two code paths for the same logical operation; (2) the event payload is limited to `key`, `oldValue`, and `newValue` ŌĆö all strings ŌĆö requiring manual JSON serialization/deserialization and making it impossible to distinguish between multiple rapid writes to the same key; (3) BroadcastChannel provides a cleaner pub/sub model with structured cloning (preserving Date objects, nested structures), typed events via the `type` field, and explicit channel naming that decouples the event bus from the storage layer. BroadcastChannel is the modern, purpose-built API for same-origin cross-tab messaging, and it is the correct choice for this architecture.

### 3.2 Auth Module ŌĆö Scope & Boundaries

The Auth Module is the gatekeeper of the entire application. It owns every aspect of the user identity lifecycle within the browser's origin: registration of new mock users, validation of login credentials against localStorage, establishment and teardown of the session flag, auto-seeding of demo users on first visit, and enforcement of the authentication boundary through route guards. This module is the sole owner of the `users` key in localStorage and the `currentUser` session flag ŌĆö no other module may read or write these keys directly; all access must flow through the Auth Module's interface.

**Registration flow.** When a new user submits the registration form, the Auth Module validates that the email is not already present in the `users` collection (duplicate detection), constructs a new user entity with obviously fake PII values (email at `@demo.local` or `@example.local`, fictional display name, plaintext demo password), persists it to the Repository Layer, and immediately establishes a session by writing the `currentUser` flag. The user is then redirected to the main chat interface without a second login step. This "register-then-auto-login" pattern eliminates friction and directly supports the activation metric target of Ōēź60% of users sending a first message.

**Login flow.** The login path validates submitted credentials against the `users` collection retrieved through the Repository Layer. On success, it writes the `currentUser` session flag and signals the UI Shell to navigate to the protected chat route. On failure, it returns a structured error that the UI renders as an inline message ŌĆö the Auth Module itself has no opinion about error presentation; it only produces the error state. The login screen also exposes pre-seeded demo users as one-click login buttons; selecting one bypasses the credential form entirely and directly establishes a session for that demo user.

**Session persistence.** On application initialization, before any UI renders, the Auth Module reads the `currentUser` flag from localStorage. If the flag is present and references a valid user entity in the `users` collection, the module hydrates the authenticated user context and the UI Shell proceeds directly to the chat interface. If the flag is absent or invalid (e.g., the referenced user was somehow removed), the module signals an unauthenticated state and the UI Shell redirects to the login screen. This check happens synchronously during bootstrap to prevent any flash of protected content.

**Logout.** Logout is a narrow operation: the Auth Module removes only the `currentUser` key from localStorage. It deliberately leaves all other data ŌĆö the `users` collection, conversations, messages, delivery statuses ŌĆö intact. This design choice enables the multi-tab demo scenario where one user logs out and another logs in on the same browser, immediately seeing the pre-seeded conversations and messages without data loss.

**Demo user seeding.** On first application load, when the `users` key is absent from localStorage, the Auth Module seeds 3ŌĆō5 demo users with fictional identities. This seeding runs synchronously and exactly once, gated by a strict emptiness check, to prevent race conditions where a manual registration could be overwritten. The seeded users are immediately available for one-click login and appear in the "New Conversation" user picker.

**Route protection.** The Auth Module provides a guard mechanism consumed by the UI Shell & Routing Module. The guard checks for the presence and validity of the `currentUser` session flag before allowing navigation to any protected route. If the guard fails, navigation is redirected to `/login`. This guard is the single, mandatory enforcement point ŌĆö there is no alternative path to protected content.

**Boundaries.** The Auth Module does NOT own: conversation data, message content, presence state, the UI layout, or the routing table. It does not know about BroadcastChannel events. It interacts only with the Repository Layer (for user CRUD) and the UI Shell (for navigation signals). It is the only module permitted to write the `currentUser` key and the `users` collection key.

### 3.3 Conversation Module ŌĆö Scope & Boundaries

The Conversation Module owns the full lifecycle of conversations ŌĆö both direct (1:1) and group ŌĆö and their membership. It is responsible for creating conversations, enforcing the canonical DM reuse rule (no duplicate 1:1 conversations between the same two users), managing group membership with admin/member role distinctions, handling group renaming (admin-only), and maintaining the `last_message_at` timestamp that drives the conversation list sort order. This module addresses the conversation management requirements from SCRUM-20950.

**Direct message creation with canonical reuse.** When a user initiates a DM with another user, the Conversation Module does not blindly create a new conversation. It first queries the Repository Layer for an existing `direct`-type conversation whose member set exactly matches the two participants. If found, it returns the existing conversation ŌĆö this is the canonical reuse guarantee. Only if no such conversation exists does it create a new one, persisting both the conversation entity and two `ConversationMember` records. This logic prevents the proliferation of duplicate DM conversations and ensures that message history is always consolidated under a single conversation.

**Group creation.** Group creation requires a name and at least two other members (total Ōēź 3 including the creator). The Conversation Module validates the member count, assigns the creator the `admin` role and all other members the `member` role, persists the conversation and membership records, and broadcasts a `conversation.updated` event via the Presence & Real-Time Module so that other tabs belonging to group members can immediately reflect the new group in their conversation lists.

**Membership management.** Only users with the `admin` role may add or remove members. When a member is removed, the Conversation Module deletes their `ConversationMember` record but preserves all messages sent up to the point of removal ŌĆö the removed member retains read-only access to conversation history. New members added after conversation creation receive access to the full message history. Both add and remove operations broadcast `conversation.updated` events to keep all tabs synchronized.

**Group renaming.** The admin-only rename operation updates the conversation's `name` field and broadcasts the change. The Conversation Module validates that the requesting user holds the `admin` role before permitting the mutation.

**Conversation list ordering.** The Conversation Module maintains the `last_message_at` timestamp on every conversation. This field is updated by the Message Module (via a cross-module interface) whenever a new message is sent to a conversation. The Conversation Module uses this timestamp to provide conversations sorted in descending order of recent activity to the UI Shell.

**Boundaries.** The Conversation Module does NOT own: message content or delivery status (that belongs to the Message Module), presence information (Presence & Real-Time Module), user authentication state (Auth Module), or UI rendering (UI Shell). It depends on the Repository Layer for all persistence operations and on the Presence & Real-Time Module for broadcasting conversation-level changes. It exposes a query interface that the UI Shell uses to populate the conversation list and a mutation interface that the UI Shell calls in response to user actions ("New Conversation" modal, group settings).

### 3.4 Message Module ŌĆö Scope & Boundaries

The Message Module is the heart of the chat experience. It owns the complete message lifecycle: composition validation, persistence orchestration, local UI update signaling, cross-tab broadcast, date-separated threading, timestamp formatting, and delivery/read receipt state management. This module is the primary driver of SCRUM-20950 and directly impacts the critical performance NFR of sub-500ms p95 message display latency.

**Message composition and validation.** Before any persistence or broadcast occurs, the Message Module validates incoming message content against two rules: (a) blank or whitespace-only messages are rejected with a structured error, and (b) messages exceeding 4,000 characters are blocked with a clear error. These validations happen synchronously in the composition pipeline, before any I/O, to provide immediate feedback to the user. Only messages that pass both validations proceed to the send pipeline.

**Send pipeline.** The send pipeline is a carefully ordered sequence designed to guarantee the reliability NFR of 100% message persistence before UI display. The pipeline proceeds as follows: (1) validate content, (2) construct the message entity with sender ID, conversation ID, content, and timestamp, (3) persist the message to the Repository Layer (which synchronously writes to localStorage), (4) update the conversation's `last_message_at` timestamp via the Conversation Module's interface, (5) signal the local UI to display the message immediately, and (6) broadcast a `message.created` event via the Presence & Real-Time Module to all other tabs. The critical ordering constraint is that step 3 (persistence) must complete before step 5 (UI display) ŌĆö this ensures that if localStorage write fails, the message is never shown to the user as "sent" when it was not actually persisted.

**Date-separated threading.** The Message Module groups messages within a conversation by calendar date. It computes date boundaries from message timestamps and inserts logical date separators ("Today", "Yesterday", or absolute dates like "Jan 15, 2025") between groups. Each individual message bubble carries its own timestamp (e.g., "10:32 AM"). This grouping is a pure presentation concern computed from the message list; it does not alter the underlying message entities or their storage format.

**Delivery and read receipt state machine.** Every sent message progresses through three states: `sent` (single checkmark ŌĆö the message has been persisted and broadcast), `delivered` (double checkmark ŌĆö the recipient's tab has received the `message.created` event and acknowledged it), and `read` (blue double checkmark ŌĆö the recipient has viewed the message in the conversation thread). The Message Module manages this state machine through `MessageDelivery` records, one per recipient per message. When a recipient's tab receives a `message.created` event, it creates a `delivered` status record and broadcasts a `message.status` event back. When the recipient views the conversation, the module advances the status to `read` (monotonically ŌĆö `last_read_message_id` only moves forward to prevent race conditions) and broadcasts another `message.status` event. In group conversations, read receipts aggregate across all members: "Read by N/M" or a list of reader names. The reciprocal read-receipt opt-out (FR-022) is enforced here: if a user has disabled read receipts, the Message Module neither publishes that user's read status nor displays others' read status to that user.

**Boundaries.** The Message Module does NOT own: conversation creation or membership (Conversation Module), user authentication (Auth Module), the BroadcastChannel transport (Presence & Real-Time Module), or UI rendering (UI Shell). It depends on the Repository Layer for message and delivery status persistence, on the Conversation Module for updating `last_message_at`, and on the Presence & Real-Time Module for broadcasting events. It exposes query interfaces (list messages for a conversation, paginated) and command interfaces (send message, mark read) to the UI Shell.

### 3.5 Presence & Real-Time Module ŌĆö Scope & Boundaries

The Presence & Real-Time Module is the nervous system of the application. It owns the BroadcastChannel transport layer, the presence heartbeat protocol, the typing indicator state machine, and all cross-tab event propagation. This module is what makes the ChatApp Demo feel "live" despite having no server ŌĆö it simulates real-time behavior entirely within the browser through inter-tab communication. It serves both SCRUM-20950 (real-time messaging features) and indirectly SCRUM-20951 (multi-tab demo experience).

**BroadcastChannel Service (embedded).** At the core of this module sits the BroadcastChannel Service, which owns the `chat-app-demo` channel. This service handles channel lifecycle (creation on app start, closure on tab unload), event serialization to JSON for transport, deserialization on receipt, and event type routing to registered handlers. It also performs feature detection: if the browser does not support BroadcastChannel, the service degrades gracefully to a no-op mode, and the UI displays an appropriate hint. The service supports five event types: `message.created`, `message.status`, `typing.start`, `typing.stop`, and `presence.update`. Each event type has a dedicated handler registered by the owning domain module (Message Module handles `message.*` events; Presence & Real-Time Module handles `typing.*` and `presence.*` events).

**Presence heartbeat protocol.** Presence is simulated through a heartbeat mechanism. The current tab broadcasts a `presence.update` event every 5 seconds containing the logged-in user's ID and an `online` status. Other tabs listen for these heartbeats and maintain an in-memory map of user presence: a user is `online` if a heartbeat was received within the last 10 seconds, otherwise they transition to `offline` with a `last_seen_at` timestamp derived from the most recent heartbeat. The heartbeat interval (5 seconds) and timeout threshold (10 seconds) are tuned to balance responsiveness (FR-015 requires status flip to online within 2 seconds of tab connection and to offline within 30 seconds of disconnection) against unnecessary BroadcastChannel traffic. The 3-second inactivity timeout on the receiving side serves as a fallback cleanup mechanism in case a tab crashes or closes without sending `typing.stop`.

**Event propagation guarantees.** The Presence & Real-Time Module makes no delivery guarantees beyond what the BroadcastChannel API provides ŌĆö events are fire-and-forget within the browser origin. There is no acknowledgment protocol, no retry logic, and no persistence of events. If an event is missed (e.g., a tab was not yet open), the receiving tab will catch up by reading the canonical state from localStorage when it loads the relevant conversation.
